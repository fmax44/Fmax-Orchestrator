# Рабочий регламент ChatGPT + Codex + MCP Orchestrator

## 1. Роли

### ChatGPT — архитектор

Формулирует цель, разбивает работу на маленькие задачи, проверяет отчёты, diff и тесты, принимает или возвращает задачу.

### Codex Desktop — исполнитель

Читает задачу из `.codex/tasks`, меняет только разрешённые файлы, запускает проверки и пишет отчёт в `.codex/reports`.

### MCP Orchestrator — диспетчер задач и проверок

Создаёт задачи, хранит статусы, читает отчёты, показывает git diff, запускает проверки, фиксирует решения архитектора.

## 2. Как начать новый проект

1. Убедиться, что проект уже является Git-репозиторием.
2. Запустить bootstrap:

```powershell
npm run bootstrap -- --project "D:\projects\some-project"
```

3. Проверить здоровье проекта через `project_health`.
4. Убедиться, что `.codex/` есть в `.gitignore`.

## 3. Как поставить задачу

Использовать `create_task`. Задача должна быть маленькой, проверяемой и с явными acceptance criteria.

Обязательно указать:

- цель;
- scope;
- out of scope;
- files allowed;
- required checks;
- формат отчёта.

## 4. Как Codex должен выполнять задачу

Codex читает `.codex/tasks/<id>-task.md`, меняет только разрешённые файлы и избегает лишних рефакторингов.

Если задача неясна или небезопасна, Codex пишет это в отчёт вместо догадок.

## 5. Как Codex должен писать отчёт

Отчёт пишется в `.codex/reports/<id>-report.md`.

Минимальные разделы:

- Summary;
- Changed Files;
- Checks Run;
- Manual Verification;
- Problems;
- Security Notes;
- Next Step Proposal.

## 6. Как ChatGPT проверяет результат

Использовать:

- `get_task_status`;
- `read_report`;
- `inspect_diff`;
- `run_tests`;
- ручной `git status --short` при необходимости.

## 7. Когда принимать задачу

Принимать через `approve_task`, если:

- цель выполнена;
- acceptance criteria закрыты;
- проверки проходят;
- diff соответствует scope;
- нет секретов и лишних файлов.

## 8. Когда возвращать на доработку

Возвращать через `reject_task`, если:

- тесты или сборка падают;
- diff выходит за scope;
- отчёт неполный;
- есть риск безопасности;
- задача выполнена частично.

## 9. Как создавать следующую задачу

После approve использовать `create_next_task`. Новая задача должна учитывать отчёт и результат предыдущей.

## 10. Что нельзя делать

- Не хранить секреты в коде.
- Не коммитить `.codex/`, `.env`, `node_modules/`, `dist/`.
- Не запускать опасные команды.
- Не менять файлы вне `Files Allowed`.
- Не добавлять лишнюю функциональность.
- Не запускать Codex Desktop автоматически в MVP-сценарии.

## 11. Чек-лист перед approve

- Отчёт прочитан.
- Diff просмотрен.
- Build/test/lint выполнены, если есть.
- Изменения соответствуют задаче.
- Нет секретов.
- Есть понятный next step.

## 12. Чек-лист перед commit

- `git status --short` понятен.
- `git diff --stat` ожидаемый.
- `.env`, `.codex`, `node_modules`, `dist` не попали в Git.
- Commit message описывает результат задачи.

## 13. Типовой цикл работы

1. `project_health`.
2. `create_task`.
3. Codex выполняет задачу.
4. Codex пишет отчёт.
5. `read_report`.
6. `inspect_diff`.
7. `run_tests`.
8. `approve_task` или `reject_task`.
9. Commit в рабочем проекте.
10. `archive_task` для завершённой задачи.
11. `create_next_task`.
