# Architecture Decision Log

## ADR-010: Почему нужна project policy

### Решение

Добавить локальный `.codex/project-policy.json`, который описывает правила безопасности конкретного проекта.

### Причина

Общие denylist-правила недостаточны для реальных проектов. Разные репозитории имеют разные приватные папки, protected files, допустимые проверки и профили запуска.

### Последствия

Orchestrator может валидировать задачу до выдачи Codex, проверять diff перед approve и выбирать безопасные doctor/smoke defaults из policy.

### Ограничения

Policy является локальным файлом внутри `.codex` и не заменяет человеческий review. Команда может решить версионировать policy отдельно, но по умолчанию `.codex` не tracked.

## ADR-013: Почему approve_task должен проходить через Review Gate

### Решение

`approve_task` должен автоматически запускать Review Gate перед изменением статуса задачи на `approved`.

### Причина

До MVP-8 архитектор вручную вызывал несколько tools: task status, report, diff, tests, policy validation и forbidden paths. Ручная последовательность полезна, но её легко выполнить неполно.

### Последствия

Approval становится единым контролируемым gate: `APPROVABLE` проходит, `NEEDS_REVIEW` требует override, `BLOCKED` требует force reason.

### Ограничения

Review Gate не заменяет человеческое решение. Он собирает evidence и блокирует очевидно опасные сценарии, но архитектор всё равно отвечает за финальное решение.

## ADR-014: Почему BLOCKED нельзя approve без force reason

### Решение

Если Review Gate вернул `BLOCKED`, обычный approve запрещён. Force approve возможен только с `force: true` и непустым `forceReason`.

### Причина

`BLOCKED` означает report missing, blockedPath diff, `.env`, `.codex`, forbidden tracked paths, failed required check или policy failure. Такие исключения должны оставлять явный audit trail.

### Последствия

Force approval записывает причину в architect decisions, а обычный approve не может случайно принять небезопасный diff.

### Ограничения

Force reason не делает изменение безопасным автоматически. Это только фиксирует осознанное исключение для последующего review.

## ADR-015: Почему strict approve должен опираться на сохранённый Review Gate provenance

### Решение

В strict workflow `approve_task` и CLI `approve` должны проверять сохранённый `lastReviewGate` из `.codex/state/tasks.json`, а не запускать новый неаудируемый review на лету.

### Причина

Если approval использует только текущий live-state, архитектор не может доказать, какой именно review-result был основанием для принятия задачи. Нужны review report path, hash и createdAt, чтобы approval ссылался на конкретное проверенное состояние.

### Последствия

Approval получает audit provenance: можно проверить, какой review был выполнен, не подменён ли report и не устарел ли он. Это делает force/override более прозрачными и пригодными для последующего разбора.

### Ограничения

Provenance не заменяет человеческий review и не гарантирует, что diff после review не изменился вне Git discipline. Поэтому strict mode дополнительно может требовать clean Git.

## ADR-016: Почему override и force должны обходить разные состояния

### Решение

`overrideReviewGate` должен разрешать только `NEEDS_REVIEW`, а `force` должен разрешать только `BLOCKED` и только с обязательным `forceReason`.

### Причина

`NEEDS_REVIEW` означает спорный, но не обязательно опасный случай: protected file, manual approval, stale review. `BLOCKED` означает нарушение барьера безопасности или отсутствие обязательного audit evidence. Смешивать эти два случая одной кнопкой рискованно.

### Последствия

Архитектор явно различает ручное подтверждение обычного предупреждения и осознанное принятие блокирующего риска. Audit trail становится понятнее, а policy легче объяснять команде.

### Ограничения

Даже с такой схемой force остаётся исключением и требует организационной дисциплины: его нельзя превращать в обычный путь approve.

## ADR-011: Почему blockedPaths важнее allowedPaths

### Решение

Если путь одновременно попадает в `allowedPaths` и `blockedPaths`, побеждает `blockedPaths`.

### Причина

Списки разрешений часто бывают широкими, например `backend/**` или `docs/**`. Запрещённые зоны должны иметь приоритет, чтобы `.env`, `.codex`, `storage/uploads`, `backups` и build artifacts не менялись случайно.

### Последствия

Policy validation может блокировать задачу или diff даже если путь формально находится в разрешённой области.

### Ограничения

Слишком широкий `blockedPaths` может потребовать ручного обновления policy, если проектная структура изменилась.

## ADR-012: Почему protectedFiles требуют manual approval

### Решение

`protectedFiles` и `manualApprovalRequiredFor` не всегда блокируют задачу, но выставляют `manualApprovalRequired: true`.

### Причина

Некоторые файлы менять можно, но только после осознанного решения архитектора: `docker-compose.yml`, миграции, storage/backups, инфраструктурные config-файлы.

### Последствия

Orchestrator может отличать полностью запрещённые изменения от изменений, требующих отдельного подтверждения.

### Ограничения

Manual approval является сигналом для процесса review, а не криптографическим запретом на изменение файла.

## ADR-008: Почему smoke должен иметь ephemeral mode

### Решение

Добавить `smoke --ephemeral` и использовать его как рекомендуемый режим ежедневных smoke-проверок.

### Причина

Legacy smoke создаёт обычную задачу, расходует нормальные task IDs и может засорять рабочую очередь. Для эксплуатационной проверки это лишний шум: smoke должен проверять инфраструктуру, но не конкурировать с реальными задачами.

### Последствия

Ephemeral smoke пишет данные в отдельный namespace `.codex/smoke/tasks`, `.codex/smoke/reports` и `.codex/smoke/state`. Обычные `.codex/state/tasks.json` и `.codex/tasks` остаются неизменными, а `git status --short` остаётся clean, если `.codex/` исключён из Git.

### Ограничения

Legacy smoke сохраняется для обратной совместимости. Он по-прежнему может расходовать обычные task IDs, поэтому для ежедневной работы нужно явно рекомендовать `--ephemeral`.

## ADR-009: Почему docker compose config output не сохраняется по умолчанию

### Решение

Docker Compose profile не выполняет `docker compose config` по умолчанию. Команда выполняется только при явном `--allow-compose-config-output`, но полный stdout/stderr всё равно не сохраняется в отчёты.

### Причина

`docker compose config` может раскрывать resolved env values из `.env` и окружения. Такие значения не должны попадать в отчёты, markdown-файлы, MCP JSON или commit history.

### Последствия

Безопасный профиль проверяет наличие compose-файла, доступность `docker compose version` и выполнение `docker compose ps`. Если `docker compose config` разрешён явно, сохраняются только pass/fail, exit code и предупреждение о чувствительности вывода.

### Ограничения

Этот режим не заменяет полный ручной аудит Docker Compose. Он нужен как безопасная эксплуатационная проверка перед работой Orchestrator в реальном проекте.

## ADR-001: Использовать файловую очередь задач

### Решение

Задачи и отчёты хранятся как markdown-файлы в `.codex/tasks` и `.codex/reports`.

### Причина

Файлы легко читать человеку, удобно проверять через Git и просто использовать без отдельной базы данных.

### Последствия

Codex Desktop может работать с задачей напрямую в проекте, а ChatGPT получает прозрачную историю работы.

### Ограничения

Файловая очередь не решает конкуренцию нескольких исполнителей и требует дисциплины при ручных изменениях.

## ADR-002: Не запускать Codex Desktop автоматически в MVP

### Решение

Orchestrator не запускает Codex Desktop сам.

### Причина

MVP должен быть простым, контролируемым и безопасным. Ручной запуск уменьшает риск неожиданных действий.

### Последствия

Пользователь явно видит, когда Codex выполняет задачу.

### Ограничения

Процесс требует ручного шага между постановкой задачи и отчётом.

## ADR-003: Сделать Git обязательным

### Решение

Рабочий проект должен быть Git-репозиторием до использования diff/review workflow.

### Причина

Git даёт проверяемый diff, историю commit и понятную границу изменений.

### Последствия

`inspect_diff`, approve и commit-review становятся надёжными.

### Ограничения

Проекты без Git нужно сначала инициализировать.

## ADR-004: Не трекать `.codex` в Git

### Решение

`.codex/` по умолчанию добавляется в `.gitignore`.

### Причина

Очередь задач, отчёты и решения являются локальной рабочей координацией, а не частью продукта.

### Последствия

В Git остаются только изменения продукта, а не служебный шум.

### Ограничения

Если команде нужна общая очередь задач, это решение придётся пересмотреть.

## ADR-005: Использовать TypeScript для MCP-сервера

### Решение

MCP Orchestrator реализован на TypeScript и Node.js.

### Причина

Официальный MCP SDK доступен для TypeScript, а строгие типы помогают держать tools и сервисы предсказуемыми.

### Последствия

Сервер легко расширять новыми tools, prompts и services.

### Ограничения

Нужны Node.js и npm-зависимости.

## ADR-006: Добавить doctor для эксплуатационной диагностики

### Решение

Добавить doctor-команду и MCP tool `doctor`, которые проверяют готовность Orchestrator и целевого проекта.

### Причина

Перед передачей задач Codex нужно быстро понять, работает ли окружение, Git, scripts и `.codex`-структура.

### Последствия

Пользователь получает один понятный readiness result: `READY`, `READY_WITH_WARNINGS` или `NOT_READY`.

### Ограничения

Doctor не исправляет проект автоматически. Он показывает состояние и ожидает явного решения пользователя.

## ADR-007: Добавить smoke без изменения бизнес-кода

### Решение

Smoke создаёт и архивирует служебную задачу только внутри `.codex`, не меняя tracked business files.

### Причина

Нужно проверять полный Orchestrator workflow в реальном проекте без риска затронуть продуктовый код.

### Последствия

Можно безопасно проверять create/read/report/diff/archive flow перед рабочей сессией.

### Ограничения

Smoke требует, чтобы `.codex/` был исключён из Git. Иначе проект может стать грязным.
